# CLAUDE.md — 本リポジトリでの作業規範

## プロジェクト
中小企業診断士試験 学習アプリ(React+Vite PWA → Capacitor iOS)。
仕様は `REQUIREMENTS.md` が唯一の正。判断に迷ったら仕様に立ち返り、勝手に要件を追加しない。

## データの扱い(最重要)
- `data/` 配下のマスターデータは**読み取り専用**。アプリ側で問題文・正解・配点を書き換えない。
- 正解・配点の改変が必要に見えたら、それはデータのバグ報告事項。`data/` を直接直さず issue 化。
- 採点ロジックは `all_correct` / `no_scoring` / 科目別合格基準(60%・足切り40%)を必ず反映。
  この3点はユニットテストなしにマージ禁止。
- `〓` は未解決グリフ。ユーザーに見せる際は頁画像(figure)への導線を必ず添える。

## 開発原則
1. **Fail Loud**: 例外は握りつぶさない。データ欠損・スキーマ不一致は起動時に検出して明示。
2. **Surgical Changes**: 差分は最小限。無関係なリファクタを同一コミットに混ぜない。
3. **Independent Critic**: 実装後、自分のコードに対して「落ちるケースはどこか」を列挙してから完了宣言。
4. **Compound Engineering**: 同じ失敗を二度したら、防止策(型・テスト・lint)を仕組みに昇格。

## コーディング規約
- TypeScript strict。`any` 禁止(やむを得ない場合は理由コメント必須)。
- データ型は `src/types/` に集約し、`data/*.json` のスキーマ(REQUIREMENTS §3.1)と一致させる。
- 日付処理はすべて JST 前提を明示(試験カウントダウンで時差バグを出さない)。
- コミットメッセージは日本語可、変更理由を1行目に。

## やらないこと
- 過去問テキストの外部送信(API・アナリティクスへの問題文送出は著作権上不可)。
- `sources/` の原本PDFのリポジトリへのコミット(容量・権利の両面)。
